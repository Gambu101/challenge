1. System Overview

The stack is a production-grade Solana transaction infrastructure system that orchestrates the full lifecycle of a Jito bundle from slot awareness through finalization. It integrates three real-time data sources (Geyser stream, Jito tip API, and the RPC commitment chain) and delegates all operational decisions to an AI agent powered by Claude claude-sonnet-4-20250514.

2. Component Architecture

GeyserStreamManager
Connects to Yellowstone gRPC (Triton/SolInfra). Subscribes to slot updates at PROCESSED commitment. Emits slot events at ~400ms intervals. Handles reconnection with exponential backoff (1s → 30s max). Provides leader window lookup and slot-wait primitives.
TipCalculator
Queries Jito's /api/v1/bundles/tip_floor endpoint for recent landed tips. Computes p25/p50/p75/p90 percentiles. 10s TTL cache prevents excessive API calls. Zero hardcoded values — all tips derived from live market data.
AIAgent (Claude)
Three decision surfaces: submission timing (slot/leader awareness), tip sizing (market + urgency), and failure retry (classification + economics). Each call provides rich structured context and receives a reasoned JSON decision. Claude claude-sonnet-4-20250514 via Anthropic SDK.
JitoBundleBuilder
Constructs [userTx, tipTx] bundle pairs. Signs transactions with wallet keypair. Fetches blockhash at CONFIRMED commitment only. Submits via Jito sendBundle JSON-RPC. Polls getBundleStatuses for landing confirmation.
ConfirmationTracker
Polls getSignatureStatus at 400ms for commitment progression. In full production pairs with Geyser transaction subscriptions for instant PROCESSED notification. Marks each lifecycle stage with slot number and timestamp.
LifecycleLogger
Appends structured JSONL to lifecycle.jsonl. Records: bundle ID, attempt number, tip amounts, slot numbers at each commitment stage, latency deltas, AI decisions and reasoning, failure classification, blockhash refresh events.

4. AI Agent Design

The AI agent uses Claude claude-sonnet-4-20250514 via the Anthropic Messages API. It receives structured JSON context and returns JSON decisions. Each decision surface is a separate function call to isolate reasoning.

Decision Surface 1: Submission Timing

Input: current slot, upcoming Jito leader window (startSlot, endSlot, leader), slot pressure (0–1).
Output: { submitNow, waitSlots, reasoning }
Reasoning example: "Leader window starts in 8 slots. Submit now to maximize propagation time — waiting risks blockhash expiry with no landing benefit."

Decision Surface 2: Tip Intelligence

Input: tip market stats (p25/p50/p75/p90), urgency level, network load, recent bundle history (landed/failed with tip amounts).
Output: { lamports, reasoning, confidence }
Reasoning example: "Recent bundles landing at p50 (5100 lam). High urgency + 0.78 pressure → p90 × 1.5 = 12400 lam to guarantee inclusion."

Decision Surface 3: Failure Reasoning & Retry

Input: failure class, failure message, attempt count, tip used, network conditions, previous attempt history, current tip market.
Output: { shouldRetry, newTipLamports, refreshBlockhash, waitSlots, reasoning, confidence }
Reasoning example: "Expired blockhash — recoverable. Blockhash must be refreshed. No tip competitiveness issue. Retry with 1.5x tip as precaution, wait 0 slots."

The agent is not a wrapper over hardcoded rules. For each decision it performs genuine reasoning: evaluating the economics of retrying, the cause of failure, whether the transaction itself is valid, and the cost-to-probability tradeoff of different tip levels.

confirmedCONFIRMED (32 votes)getSignatureStatus polling600–2100ms from processed
finalizedFINALIZED (supermajority)getSignatureStatus polling8–14s from confirmed

All four stages are recorded with slot number and Unix timestamp. Latency deltas (submitToProcessMs, processToConfirmMs, confirmToFinalizeMs) are computed and stored for post-hoc network health analysis.

The processed → confirmed delta is the most useful real-time network health signal: values above 1.5s indicate validator vote latency stress. Values above 3s suggest fork contention or staked validator lag.

6. Failure Handling

Failure ClassRoot CauseAI Response Strategy
expired_blockhash Blockhash > ~150 slots old at inclusion time Always refresh blockhash. Modest tip increase (1.5×). Retry immediately.
fee_too_low Transaction priority fee insufficient Significant tip increase (2–3×). Retry at next leader window.
compute_exceeded CU budget exhausted by transaction Do NOT retry (tip won't fix a CU problem). Flag for manual review.
bundle_failure Jito rejected bundle (race, invalid tx, etc.) Moderate tip increase (1.5–2×). Retry. Refresh blockhash if > 100 slots old.
leader_skip Scheduled Jito leader skipped slot Resubmit with same or modestly higher tip. No blockhash refresh needed unless stale.
simulation_failure Transaction would fail on-chain Never retry. Transaction itself is broken.

All retry decisions are made by the AI agent, not hardcoded. The agent considers attempt count, tip market conditions, and the economic viability of continuing before deciding.

7. Infrastructure Decisions

Blockhash commitment: always CONFIRMED, never FINALIZED

A finalized blockhash is ~31 slots old at fetch time. Using it wastes 12–13 seconds of the 60-second validity window before submission even begins. We always fetch at confirmed commitment — 32 validator votes ensures stability while preserving maximum lifespan.

Stream-first confirmation, not RPC-only

RPC polling at 400ms intervals adds up to 200ms average latency for detecting processed stage. In production, Geyser transaction subscriptions emit the event instantly when the block is produced — typically reducing time-to-detect by 150–300ms. The ConfirmationTracker is built to accept Geyser events as the fast path, with RPC as fallback.

Jito tip account rotation

Jito exposes 8 tip accounts. We select randomly per bundle to distribute load and avoid contention. All 8 are economically equivalent — they funnel to Jito's validator program.

Exponential backoff on stream reconnection

Geyser streams can drop due to network hiccups or endpoint maintenance. Reconnection uses 1s → 2s → 4s → ... → 30s backoff to avoid hammering the endpoint during outages.

AI agent context isolation

Each AI decision call is stateless and self-contained. All relevant context is serialized into the prompt. This avoids complex state management and makes each decision auditable from the lifecycle log alone.
